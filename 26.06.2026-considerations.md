# Architecture & Design Document: Procedural Swiss Driving Project

This document serves as a comprehensive overview of the aesthetic, technical, and architectural strategies for developing a browser-based, procedurally generated driving game focused on the Swiss pre-Alps.

---

## I. Visual Identity: Abstracting the Pre-Alps

To bypass the limitations of procedural satellite imagery and emphasize gameplay loops, the visual direction deliberately avoids photorealism. The aesthetic leverages high contrast, stark geometry, and stylized rendering to capture the isolating, dramatic atmosphere of mountain passes.

* **Excavated Granite Roads & Terrain Carving:** To maximize driving realism and physics stability, road splines do not merely drape over the topological data; they actively alter it. The procedural generation engine will carve the terrain mesh itself into a flat, drivable surface. As a road spline is plotted, the engine forces the underlying terrain vertices to flatten to the road's exact elevation, creating a perfectly smooth physics collider for the vehicle tires.
* **The "Blasted Rock" Aesthetic:** This carving process algorithmically generates sheer vertical cuts on the uphill side of the spline and sharp drop-offs on the downhill side. These newly generated steep faces are assigned a flat, stark slate-grey material to emulate blasted rock, heightening the tension of navigating high-speed mountain passes in performance vehicles.
* **Elevation-Based Color Banding:** Terrain color is dictated by a simple shader evaluating altitude and slope angle. Flat valleys use deep, rich greens, slopes exceeding 60 degrees expose stark granite grey, and high altitudes transition into pure white snowcaps.
* **Atmospheric Depth:** A stylized, thick volumetric fog settles in the valleys and over the lakes. Breaking through the cloud layer during an ascent visually sells the massive vertical scale of the environment.
* **Mass Abstracted Vegetation:** Forests are represented by dense clusters of highly abstracted, low-poly pine trees (e.g., dark green pyramids) explicitly mapped to the lower elevations, creating a stark contrast against the barren rock faces above.

## II. Data Pipeline: Swisstopo Integration

The procedural generation relies on deterministic geographic data rather than randomized noise functions, providing an accurate structural foundation for the stylized graphics.

* **swissTLM3D (Topographic Landscape Model):** This 3D vector dataset dictates biome boundaries. Land cover polygons (e.g., "Forest", "Rock") strictly control where tree clusters spawn or where granite textures override the terrain. Point-object coordinates can be used to spawn specific POIs like caves directly beside the road.
* **swissSURFACE3D (Vegetation Height Model):** LiDAR-derived raster data determines the exact Z-scale of the procedural tree meshes. Valley forests will algorithmically match real-world 30-meter canopy heights and dynamically shrink into brush as the elevation increases.

## III. Open-Source Map Architecture

To support community-driven development and manual refinement of the procedural generation, the map operates on a modular, version-controllable system.

* **Dual-Layer Chunk Streaming:** The world is divided into a grid (e.g., 1km x 1km tiles). If no custom data exists for a tile, the engine defaults to Layer 0 (pure procedural generation via Swisstopo data). If a custom prefab exists, it loads Layer 1 (manual overrides).
* **Git-Centric Workflow:** Map overrides are stored as lightweight, human-readable data structures (JSON or plain text nodes). This allows developers to branch, modify, and merge road segment improvements without binary conflicts.
* **Developer-Focused Tooling:** To lower the barrier to entry, editing tools should mimic a clean, IDE-inspired interface with high-density layouts, dark mode, and monospace typography, making it intuitive for technical contributors to adjust splines and vertices in-browser.
* **Locked Primitive Library:** To maintain a unified aesthetic, contributors do not import custom meshes or textures. They act as level designers, adjusting the parameters, scale, and placement of the core stylized assets.

## IV. Core Rendering Mechanics

Supporting massive view distances (e.g., viewing the distant Alps from a valley floor) requires specific engine architecture to prevent precision loss and manage rendering throughput.

* **Floating Origin:** To circumvent 32-bit floating-point precision limits, the camera remains anchored near absolute zero (0,0,0). As the vehicle crosses chunk boundaries, the entire coordinate system mathematically shifts in the opposite direction, preventing distant geometry from jittering.
* **Quadtree Level of Detail (LOD):** Terrain and assets are dynamically subdivided based on camera proximity. The immediate vicinity features high-density geometry, while distant mountains are rendered as massive, low-resolution polygon blocks, seamlessly updating as the player navigates.

## V. Browser-Based Tech Stack

The entire application is designed to run natively in modern web browsers using WebGL/WebGPU, removing client download friction and supporting a collaborative ecosystem.

* **React Three Fiber (R3F) + Next.js:** A component-driven approach that pairs perfectly with modern web infrastructure. Utilizing Next.js for routing and server-side logic alongside R3F makes implementing modular chunking and managing the data flow highly intuitive. The Rust-based `@react-three/rapier` handles tight arcade-sim physics.
* **Babylon.js:** A heavy-duty, web-first engine featuring native support for massive scene management, frustum culling, and dynamic LODs. It integrates the industry-standard Havok physics engine, ideal for complex vehicle suspension dynamics.
* **PlayCanvas:** An open-source WebGL engine featuring a real-time, collaborative visual editor. This is highly effective for a core group of technical contributors to concurrently edit scenes, tweak lighting, and adjust collision bounds within the same browser session.
* **Data Delivery via Vector Tiles:** To prevent browser crashes, the massive Swisstopo datasets cannot be sent directly to the client. A backend tile server (or pre-compiled JSON CDN) will compress the data into lightweight vector tiles requested dynamically based on the camera's (X, Y, Zoom) coordinates.
